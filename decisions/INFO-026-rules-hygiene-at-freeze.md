# INFO-026 — Гигиена правил при каждом freeze

## Слова CEO (verbatim)

> Гигиена правил (от CEO, в release-чеклист как постоянный шаг): при
> каждом freeze — deletion test по скиллу и knowledge (из
> pocock-чеклиста): для каждого правила/пункта — «что конкретно
> сломается при удалении?»; нет ответа с прецедентом → кандидат в архив
> (правила получают свой NOT-DOING с триггером возврата). Плюс проверка
> слоёв: каждое правило несёт пометку «кто читатель»
> (формулятор/исполнитель/строитель харнесов/инсталлятор) — правило без
> адресата = правило для всех = подозреваемый в налоге на чтение.
> Метрика: суммарный объём скилла+обязательных знаний по ролям — не
> растёт быстрее числа закрытых классов задач.

## Interpretation

1. **Deletion test при каждом freeze** — не разовая уборка, а постоянный
   шаг чеклиста: каждое правило скилла и каждый пункт knowledge отвечает
   «что конкретно сломается при удалении?» С ПРЕЦЕДЕНТОМ (цитата
   INPUT/VERDICT/run.log, где правило сработало). Нет ответа → кандидат
   в архив, оформляется как NOT-DOING правила с триггером возврата
   (симметрия NOT-DOING уровня спавна, INFO-018).
2. **Слои читателей**: каждое правило получает пометку адресата —
   формулятор / исполнитель / строитель харнесов / инсталлятор. Правило
   «для всех» — подозреваемый в налоге на чтение (каждый субагент платит
   контекстом за чужие правила). Разметка выполняется при первом freeze
   с этим шагом, дальше — поддерживается.
3. **Метрика роста**: объём(скилл + обязательные знания по ролям) растёт
   НЕ быстрее числа закрытых классов задач — рост без новых классов =
   осадок (sediment, Pocock п.2). Считается при каждом freeze, тренд
   пишется в release-чеклист.
4. Первое исполнение шага — следующий freeze (v0.2): текущий v0.1 уже
   подтверждён сводкой; ретроактивно не переоткрывается. В v0.1 шаг
   ВНЕСЁН как постоянный (раздел §4 RELEASE-v0.1.md — шаблон каждого
   будущего freeze).
