## Интерпретация (закрытые решения)

1. **Лендинг вне дерева.** Переезд `products/landing-v0/ → marketing/`
   отменён; лендинги остаются заверсионированными в `products/landing-v0/`
   с пометкой «frozen». qroky.com = DNS-редирект на GitHub-репо —
   **HUMAN-TASK CEO** (строка в панель и в launch-чеклист).
2. **Замороженные артефакты с триггером.** HN/Reddit-черновики и
   LinkedIn-пост НЕ строятся и НЕ полируются в этом дереве; триггер
   разморозки — дословно: «после 2–3 волн тёплой раздачи». Это NOT-DOING
   уровня дерева, не отказ.
3. **README — единственная витрина.** H1 расширяется блоком (verbatim
   смысл): «early access, built in the open, issues welcome».
4. **Новый артефакт — сообщение-передача** для мессенджеров: 2–3 абзаца,
   EN/RU/RO; содержание: что это, зачем нужно мнение получателя, ссылка,
   строка «15 минут; сломается — пиши мне»; пригодно для пересылки третьим
   лицам без редактирования.
   *Консистенси-заметка (INFO-027):* правило EN-only (INFO-021 §3.3)
   покрывает ПУБЛИЧНОЕ лицо (README/лендинг/посты). Сообщение-передача —
   артефакт личного канала (мессенджер, тёплая сеть), адресован конкретному
   человеку → RU/RO-версии легитимны по той же логике, что пилотные
   RO-материалы. Не коллизия — граница классов; зафиксирована здесь.
5. **Канал обратной связи:** GitHub issues (вход) + support-инстанс
   (`~/Work/qroky-support/`, INFO-030 п.5) драфтит ответы, отправка — только
   рукой CEO (E2). README указывает на issues.
6. **Волновая раздача** — в launch-чеклист: волна = 2–3 человека, интервал
   между волнами — дни (петля обратной связи успевает отработать до
   следующей волны).
