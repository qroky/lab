# INFO-044 — «Решить проблему, а не подсказывать»: команда qroky на машине

- **Тип:** INFO (директива CEO + продуктовое решение)
- **Дата:** 11.07.2026

## Формулировка CEO (дословно)

> Я думаю, что мы должны решить эту проблему, а не начинать подсказывать,
> дойти и что делать.

Контекст: после SEPARATION владелец дважды упёрся в «команда зависит от
того, где лежит клон» (No such file: framework/qroky.sh; cd в снесённый
~/work/qroky). Агент отвечал подсказками путей — это симптом: **если
владельцу нужна подсказка «где что лежит», это находка продукта, не повод
подсказывать** (принцип записывается как продуктовый).

## Решение (ATOM-131, kit v0.4.1)

1. **Команда `qroky` в PATH** — установка кладёт крошечный ланчер в
   `~/.local/bin/qroky` (+ строка PATH в профиль шелла, только если
   ~/.local/bin не в PATH, маркер-блоком). Дальше НАВСЕГДА и из любого
   места: `qroky update`, `qroky uninstall`, `qroky start`-подсказка.
   Клон и ~/.qroky/kit — деталь реализации, владелец их не видит.
2. **Вход без клона:** однострочник через curl с GitHub raw
   (`bash <(curl -fsSL https://raw.githubusercontent.com/qroky/framework/main/qroky.sh) install`)
   — README и handoff-сообщение получают ЕДИНСТВЕННУЮ команду входа;
   git clone исчезает из пользовательского пути.
3. **Финал/README/uninstall** говорят только «qroky update / qroky
   uninstall» — никаких путей.

## Периметр (явно, для I3)

Ланчер = ТРЕТИЙ machine-wide файл (`~/.local/bin/qroky`; плюс возможная
маркер-строка PATH в shell-профиле). Основание — эта директива CEO
(«решить проблему»); след: финал называет файл, uninstall удаляет его
вместе с остальным. I3-исключение в обеих копиях скилла дополняется
Amended-строкой (INFO-044). Если CEO против расширения периметра — ланчер
убирается, остаётся только curl-вход (сказать словом).
