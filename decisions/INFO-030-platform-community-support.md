## Interpretation

1. Модель — внутренняя переменная MT-маршрутизации, никогда предмет
   выбора пользователя. Проверка kit'а: вопроса о модели/тарифе в
   интервью НЕТ (подписка = login-check) — правило превентивное, в
   ядро-дисциплину новых фич. INTERFACE.md — карта границы для форкеров;
   регресс-тест смены версии модели — шаг 7 freeze (эталонный атом,
   сравнение поведения по verify-критериям).
2. CONTRIBUTING/forking — новый deliverable ATOM-120 (H10): BDFL (CEO),
   правки через issues/PR → touch-очередь (внешний вход в тот же
   конституционный цикл, не мимо него), форк приветствуется, Apache-код /
   защищённое имя — форки переименовываются.
3. Бэкап — восьмой пункт интервью (владелец списка расширил его сам):
   формулировка пользой, «да» не требует знать слово GitHub (gh auth
   flow ведёт за руку, как BotFather); restore = clone; приватный репо
   ПОЛЬЗОВАТЕЛЯ. Реализация — ATOM-102 (kit v0.1.1, ребёнок ATOM-100 в
   остатке конверта). G2 сухой прогон CEO — ПОСЛЕ 102 (интервью меняется).
4. Дисклеймер: «черновики и анализ; юридические/финансовые/медицинские
   решения и подписи — только человек; not professional advice» — в
   CONSENT.ro/.ru (правка с provenance-строкой Amended) и в
   distribution/README (ATOM-102) и корневой README FAQ (ATOM-120);
   ответственность за отправку внешним сторонам — на владельце всегда
   (E2 как юридический факт — уже механика Tier C, теперь и юридическая
   строка). FAQ «что если исчезнем»: всё локально, open source, работает
   без нас, форкается (ATOM-120 H1 FAQ).
5. Support-инстанс: скаффолд создан сейчас (`~/Work/qroky-support/`):
   mission.md (два «зачем» из этой директивы), runbook v0 (вход — GitHub
   issues qroky/framework через gh read-only; выход — драфты в drafts/,
   отправка — ТОЛЬКО рукой CEO первые недели = E2-гейт; автономия
   расширяется по maturity ответов, INFO-014-механика на знаниях
   поддержки). Прообраз multi-user: один инстанс обслуживает многих,
   Tier C на выходе.
6. Outcome-базлайн: скилл §4 получает I4 — финал первого разговора
   обустройства фиксирует «точку А» файлом `<project>/qroky/baseline.md`
   (что есть / чего нет / цель на горизонт, дословно подтверждённые);
   кейс/отчёт = diff базлайна и текущего; «точка Б» — по горизонту
   (heartbeat-строка ближе к концу пилота). Родня mission.md (INFO-011):
   mission = зачем, baseline = откуда.
