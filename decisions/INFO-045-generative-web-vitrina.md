## Формулировка CEO (дословно)

> Генеративная веб-витрина проекта: третья голова над той же шиной (рядом с
> сессией и Telegram). Команда qroky dashboard → система рендерит и держит
> свежим локальный статический dashboard/index.html из данных шины
> (status.yaml, NARRATIVE.md, RESULT'ы, cost-учёт). Форма интерфейса НЕ
> фиксированная — она выводится из типа проекта и эволюционирует вместе с
> ним: бизнес → воронка/статусы/юнит-экономика; креатив → галерея/этапы/
> варианты; книга → структура глав/прогресс; сад → карта участка/рендеры/
> смета. UI — производный продукт проекта, как роли — производные вопроса
> (F2-принцип, применённый к интерфейсу).
>
> Границы v1 (жёсткие): Строго read-only: витрина показывает, решения живут
> там, где паритет уже построен — Telegram-кнопки и сессия. Никаких
> действий из браузера. Без серверов и портов: файл + браузер (open
> dashboard/index.html), обновление по heartbeat или по команде. Ноль новой
> security-поверхности, работает офлайн. Экономика: рендер S/M-tier из уже
> существующих данных шины, бюджет витрины ≤5% subtree_cost проекта (как
> нарратив — прозрачность из побочных продуктов).
>
> Ценность (для приоритизации): Интерфейс для нетехнических пользователей
> (визуальный «сайт своего проекта» — снимает bash-стену окончательно);
> демо-артефакт для показов (Влад/Mihai/фаундеры — страница вместо
> терминала); зародыш multi-user наблюдения — двое смотрят один проект
> (фаундер+ментор, CEO+CFO) — первая ступень enterprise-этажа.
>
> North Star интерфейса (зафиксировать в бэклоге, НЕ исполнять): Целевая
> форма продукта — «разговор + живая поверхность»: пользователь говорит,
> рядом растёт витрина его проекта. Чат остаётся платформенным (Claude
> Code / Telegram — там уже существует паритет решений и вся инфраструктура
> диалога); наша поверхность живёт рядом (второй экран/вкладка/split), не
> вместо. Свой чат-клиент НЕ строим — записать в NOT-DOING с обоснованием:
