## 1. Вердикт одной строкой

Витрина v1 — это **один статический файл `dashboard/index.html` на проект**,
который собирается **детерминированным скриптом** из шины (status.yaml,
NARRATIVE.md, RESULT'ы, cost-учёт, decisions/) по **шаблону, который один раз
генерируется из типа проекта** M-tier'ом и дальше живёт как обычный продукт
проекта. LLM в ежедневном цикле трогается минимально (S-tier, только
plain-language строки), поэтому поддержание дёшево и в ≤5% помещается с
запасом. Строить целесообразно; конверт постройки — класса ATOM-110/111.

## 2. Состав v1 (жёсткие границы INFO-045 — подтверждены, не расширены)

Границы дословно из INFO-045, все четыре приняты как инварианты v1:

1. **Строго read-only.** Витрина показывает; решения живут там, где паритет
   уже построен — Telegram-кнопки и сессия. Никаких действий из браузера.
   В UI это отражено явно: блок «Ждёт тебя» показывает pending-гейты и
   говорит, ГДЕ ответить («ответь в Telegram или скажи в сессии»), но не
   даёт кнопок.
2. **Без серверов и портов.** Файл + браузер: `open dashboard/index.html`.
   Никакого localhost, никаких демонов кроме уже существующего heartbeat.
3. **Ноль новой security-поверхности, офлайн.** Self-contained HTML: инлайн
   CSS, никаких CDN/шрифтов/аналитики/fetch. Это инвариант, который
   enforce'ится негативным тестом в харнесе (см. §7), не доверием.
4. **Экономика: рендер S/M из существующих данных шины, ≤5% subtree_cost.**
   Механизм enforcement — в §6.

**Что входит в v1 (поставка):**

| # | Компонент | Что это |
| :- | :--- | :--- |
| 1 | `qroky dashboard` — команда | Рендер сейчас: собрать свежий index.html из шины и открыть в браузере |
| 2 | Хребет-рендерер (bash, паттерн `showcase/render.sh` ATOM-072) | Детерминированная сборка: status.yaml → статусы, RESULT'ы → «что сделано» + себестоимость, decisions/ → «ждёт тебя», TEAM-сводка — уже специфицирована (team-summary-spec.md) |
| 3 | Шаблон-генератор (M-tier, редкий) | Один раз на проект: из типа проекта + mission.md выводит `dashboard/template.html` + `dashboard/profile.yaml` (маппинг секций на источники шины) |
| 4 | Plain-language дайджест (S-tier, ежедневный) | 3–6 строк «за сутки» из хвоста NARRATIVE — единственный LLM-вызов регулярного цикла |
| 5 | Heartbeat-хук | Рендер как шаг существующего пульса (Mon–Fri 09:07) + честный штамп свежести на странице |
| 6 | Харнес | Негативные тесты: ноль сетевых ссылок, ноль форм/действий, рендер на пустой/битой шине падает громко, а не молча |
| 7 | Два стартовых типажа шаблона | «бизнес» и «универсальный» (chassis без domain-слоя); сад/креатив/книга — по мере появления реальных проектов этих типов |

## 3. Механика `qroky dashboard`: рендер и свежесть

**Двухслойная архитектура — редкое генеративное / частое механическое.**
Это ключевое решение study, оно делает экономику сходящейся:

- **Слой A — форма (генеративный, редкий).** При первом `qroky dashboard`
  (или при смене типа проекта) M-tier читает mission.md, тип проекта и
  структуру шины и порождает шаблон: `dashboard/template.html` (скелет с
  плейсхолдерами `{{...}}`) + `dashboard/profile.yaml` (какая секция откуда
  берёт данные). Шаблон — обычный продукт проекта: лежит в репо, виден
  человеку, эволюционирует явными правками, а не перегенерацией втихую.
  Стоимость ~40–60k токенов, случается раз-два за жизнь проекта.
- **Слой B — наполнение (механический, частый).** Bash-рендерер (прямой
  наследник `showcase/render.sh` из ATOM-072 — прецедент уже работает)
  подставляет в шаблон текущие данные шины: статусы атомов, стоимость,
  TEAM-сводку, pending-решения. LLM-токенов — ноль. Плюс один маленький
  S-tier вызов на plain-language дайджест «за сутки» (та же логика, что в
  heartbeat-prompt.md: без жаргона системы, честное «не знаю» вместо
  выдумки).

**Триггеры свежести (оба из INFO-045, оба в v1):**

1. **По команде** — `qroky dashboard`: рендер сейчас + `open` в браузер.
   Это основной путь «показать Владу/Mihai страницу вместо терминала».
2. **По heartbeat** — существующий пульс (launchd, будни 09:07) получает
   шаг «отрендерить дашборд». Никакого нового демона, никаких watcher'ов.

**Честность свежести вместо live-обновления.** Раз нет сервера — нет и
push-обновлений, и мы это не маскируем: страница несёт штамп «данные шины
на DD.MM HH:MM» в шапке; страница старше 24ч в будни показывает жёлтую
плашку «витрина не обновлялась — пульс мог умереть» (аналог dead-man
правила heartbeat README). Мёртвый сторож = честно устаревшая страница,
а не тихо-ложно-свежая.

**Где живёт шаблонизация.** В воркспейсе проекта: `dashboard/template.html`,
`dashboard/profile.yaml`, `dashboard/index.html` (рендер, gitignored как
`out/`), `dashboard/assets/` (фото/картинки, кладёт человек — см. риск №6).
Рендерер-скрипт — один на машину, в ките (`~/.qroky/kit`), как весь runtime.

## 4. Форма UI из типа проекта (F2-принцип: UI — производный продукт)

Как роли — производные вопроса, так витрина — производная проекта. Механика:

**Инвариантный chassis (одинаков для всех типов, весь из шины):**
- шапка: имя проекта, миссия одной строкой (из mission.md), штамп свежести;
- «Сделано / В работе / Ждёт тебя» (status.yaml + decisions/);
- «Команда работала» — TEAM-сводка (инвариант INFO-007: роли, перспективы,
  где спорили, чем кончилось; спека и рендер уже существуют в ATOM-072);
- строка себестоимости (cost-line-spec.md — тоже уже существует);
- дайджест «за сутки» plain language.

**Domain-слой (выводится из типа, разный):**

| Тип проекта | Ведущая метафора | Секции domain-слоя |
| :--- | :--- | :--- |
| Бизнес | воронка | воронка сделок/статусы, юнит-экономика, pipeline-карточки |
| Креатив (бижутерия) | галерея | галерея вариантов, этапы коллекции, материалы/смета, варианты A/B/C на выбор |
| Книга | оглавление | структура глав с прогрессом, знаки/слова, арки |
| Сад | карта | карта участка, этапы сезона, смета работ и посадок |

Шаблон-генератор (слой A) выбирает метафору по типу проекта и раскладывает
domain-секции; chassis подшивается всегда. Эволюция: тип не перечисляется
раз и навсегда — когда проект мутирует (сад-проект вырос в «ландшафтный
бизнес»), человек говорит это словами, слой A перегенерирует шаблон, старый
уходит с `-superseded-` суффиксом (дисциплина OD7/VP17 — шаблоны не
правятся молча). Оба эскиза в workspace/ иллюстрируют ровно эту пару
«общий chassis + разный domain-слой».

**Откуда данные domain-слоя.** Chassis весь из шины. Domain-слой — из
продуктов проекта (RESULT'ы, artefact'ы атомов: смета — продукт атома
«смета», карта — продукт атома «план участка»). Чего в продуктах нет —
витрина честно показывает пустую ячейку «ещё не построено», НЕ выдумывает.
Это то же правило честности, что в heartbeat-prompt («не знаю» лучше ложной
уверенности), перенесённое в UI.

## 5. Что НЕ входит в v1

- **Действия из браузера** — никакие: ни кнопок решений, ни форм, ни
  комментариев (граница INFO-045 дословно).
- **Собственный чат-клиент** — NOT-DOING, зафиксирован в INFO-045,
  здесь не пересматривается. Обоснование дословно: «(а) война на чужом
  этаже против платформ (Claude Code/Telegram/Cursor делают чат сотнями
  инженеров и отдают бесплатно; наша ценность — контур ответственности, не
  мессенджер — per Codex-review-2); (б) решения из браузера =
  auth/идентичность/подпись рисков = security-этаж, который строится только
  под платящего enterprise-клиента. Триггер пересмотра: первый клиент,
  платящий за веб-решения/мульти-пользовательский доступ.» North Star
  «разговор + живая поверхность» — строка бэклога, не работа v1.
- **Серверы, порты, watcher-демоны, live-reload** — только файл + браузер.
- **Сводная витрина нескольких проектов** — v1 по-проектная (см. риск №2).
- **Генерация картинок** (рендеры сада, фото украшений) — витрина
  показывает то, что человек положил в `dashboard/assets/`, и рисует
  инлайн-SVG схемы (карта участка) из данных; image-модели не зовутся.
- **История/архив состояний** — только текущее состояние (как status.yaml:
  current state only); машина времени — не v1.
- **Мобильное приложение / PWA / шаринг по ссылке** — файл локален;
  multi-user наблюдение остаётся «двое смотрят один экран/один файл».
- **Новые типажи шаблонов впрок** — только под реальные проекты (YAGNI;
  два эскиза study — иллюстрация формы, не поставка шаблонов).

## 6. Экономика

### 6.1 Стоимость ПОСТРОЕНИЯ v1

Калибровка — maturity-блок `roles/pilot-toolsmith.md` (10 прогонов, реальные
атомы кита 68–190k, точность конвертов 0.40–1.0):

| Аналог | Реальная цена | Почему сопоставим |
| :--- | :--- | :--- |
| ATOM-072 (telemetry+showcase рендереры) | ~194k | тот же жанр: рендер шины в фаундер-читаемый артефакт |
| ATOM-111 (project router) | ~190k | сопоставимая связность: скрипты + харнес + интеграция с существующим runtime |
| ATOM-105/106 (кит-атомы поменьше) | ~68–102k | нижняя граница жанра |

Оценка v1 (рендерер + шаблон-генератор + 2 типажа + heartbeat-хук + харнес
+ README): **исполнитель ~160–190k реальных** — это верх жанрового
диапазона (два слоя + генеративная часть), **slim-verify ~70–90k**
(калибровка: verify кит-атомов 88–128k). Конверт с запасом по фактической
точности роли (медиана ~0.85): **executor ~220k, verify ~90k, дерево
~310k hard stop**. Одно дерево, один toolsmith (тёплый, 10 прогонов) —
резать на части дороже, чем делать целиком (прецедент ATOM-101).

### 6.2 Стоимость ПОДДЕРЖАНИЯ (рендер на heartbeat)

Реальные объёмы шины rpf (замер 11.07.2026, `wc -c`):

| Источник | Объём | В токенах (~4 байта/токен) |
| :--- | :--- | :--- |
| `products/*/status.yaml`, все 7 продуктов | 16,179 байт | ~4.0k |
| status.yaml активного продукта (pilot-prerequisites, самый толстый) | 4,959 байт | ~1.2k |
| NARRATIVE.md живого дерева | 5.6–21k байт | ~1.4–5.3k |
| RESULT.md, средний (372,745 байт / 65 файлов) | ~5.7k байт | ~1.4k |

**День средней активности** (по фактическому ритму rpf: 1 живое дерево,
2–4 смены статусов, прирост NARRATIVE ~2–4k байт, 0–1 новый RESULT):

- Слой B (bash-рендер): status.yaml + хвост NARRATIVE + свежие RESULT'ы →
  **0 LLM-токенов** (детерминированная подстановка).
- S-tier дайджест «за сутки»: вход = хвост NARRATIVE (~1.5k) + дельта
  status.yaml (~0.5k) + свежий RESULT summary (~1.5k) + инструкция+шаблон
  строк (~1.5k) ≈ **~5k вход**; выход 3–6 строк ≈ 0.3k; системный оверхед
  вызова ~2k. **Итого ~7–8k токенов/рендер, 1 рендер/день ≈ 8k
  токенов/день.** Пик (день закрытия дерева, толстый RESULT + TEAM-сводка):
  ~15k. Команда `qroky dashboard` руками — те же ~8k за вызов, но слой B
  без слоя S (данные те же, дайджест не пересочиняется) — ~0.
- Слой A (перегенерация шаблона): ~40–60k, раз-два за жизнь проекта —
  амортизируется в ноль на день.

**Проверка ≤5%:** живой проект средней активности по факту rpf жжёт
~200k–2.7M за дерево (ATOM-007 subtree ~2.76M за день пика; тихие дни —
десятки тысяч). 5% от скромных 200k/день = 10k/день ≥ наших ~8k. На тихих
днях (шина не менялась) рендер обязан **скипаться** (mtime-чек до любого
LLM-вызова) — тогда витрина не тратит ничего и не съедает 5% на пустом ходу.

**Enforcement ≤5%:** рендерер пишет собственный расход строкой в тот же
cost-учёт, что все атомы (прозрачность по образцу cost-line-spec); перед
S-вызовом сравнивает кумулятив витрины с 5% от subtree_cost из status.yaml.
Превышение → слой B рендерит без дайджеста + жёлтая плашка на странице
«дайджест выключен: витрина упёрлась в свой 5%-бюджет» — деградация
громкая и видимая, не тихая (метод-дефолт toolsmith: never silently skips).

## 7. Риски и открытые вопросы

1. **Где определяется тип проекта — intake или эволюция?** Study-позиция:
   первично — на intake (два «зачем» + mission.md уже дают материал), поле
   `project_type` в profile; эволюция — явным словом человека → регенерация
   шаблона с superseded-следом. Открыто для CEO: добавлять ли вопрос типа в
   интервью install.sh (сейчас семь вопросов — INFO-035-класс ограничение)
   или выводить тип молча из mission.md без вопроса.
2. **Несколько проектов на машине: по-проектная или сводная?**
   v1 — строго по-проектная (`<workspace>/dashboard/index.html`). Прецедент
   уже решён для Telegram (ATOM-111: реестр `~/.qroky/registry`, merged
   digest) — сводная страница-оглавление по реестру напрашивается как
   дешёвый v1.1, но в v1 не тянем (YAGNI до второго реального проекта на
   машине нетехнического пользователя).
3. **Enforcement ≤5% опирается на самоотчёт.** Cost-счётчики в status.yaml
   пишутся самими агентами; витрина сравнивает себя с ними. Ложь в
   счётчиках ломает и ограничитель. Митигация v1: расход витрины — отдельной
   строкой, проверяемой глазами; строже — вопрос волны №2.
4. **Свежесть = жизнь heartbeat.** Ноут спал в 09:07 — витрина вчерашняя.
   Митигация: штамп даты + жёлтая плашка «старше 24ч» (dead-man аналог).
   Остаточный риск: человек показывает страницу гостю, не заметив плашку.
5. **Генеративный HTML = потенциальная сетевая утечка.** Шаблон порождает
   LLM; одна `<img src="https://...">` — и «ноль security-поверхности»
   нарушен. Митигация — не доверие, а тест: харнес-grep на `http(s)://` в
   src/href (кроме якорей), `<script src`, `<form>`, `fetch(` — красный =
   шаблон не принимается. Плюс `<meta http-equiv="Content-Security-Policy">`
   c `default-src 'none'` как ремень безопасности в самом файле.
6. **Данные domain-слоя не в шине.** Карта участка, фото украшений, смета —
   это продукты атомов проекта, их формат никем не стандартизован. v1
   решает конвенцией «витрина показывает то, что есть, и честно показывает
   дыры», но вопрос «должен ли intake сад-проекта сразу заказывать
   машиночитаемую смету» — открыт (связка с vertical-leak чеком:
   конституция домен-фри, домен объявляет параметры).
7. **Жаргон-утечка в plain-language дайджест.** S-tier дешевле, но чаще
   протаскивает «атом/verify/гейт» в текст фаундеру. Митигация: тот же
   запрет-словарь, что в heartbeat-prompt, + негативный grep в харнесе;
   если S систематически фейлит — M на дайджест (+~3–5k/день, всё ещё в 5%).
8. **Владение шаблоном.** Шаблон — продукт проекта (git-tracked) или
   локальный артефакт (gitignored, как `out/`)? Study-позиция: template +
   profile — в git (это форма проекта, она заслуживает истории и verify);
   `index.html` — рендер, gitignored. Открыто: не испугает ли
   нетехнического фаундера HTML-файл в его «понятном» репо.

## 8. Рекомендация

Строить в волну №2 по этому вердикту: одно дерево, тёплый pilot-toolsmith,
конверт ~310k, двухслойная механика (генеративный шаблон редкий /
механический рендер частый), стартовые типажи «бизнес» + «универсальный»,
harness-инварианты офлайновости и read-only как негативные тесты. Ежедневная
цена (~8k токенов) укладывается в ≤5% с запасом и падает до нуля в тихие
дни. Главный нерешённый вход перед стройкой — вопрос №1 (тип проекта на
intake): он трогает интервью install.sh и потому хочет решения CEO до G1.
