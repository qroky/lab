# Уроки полевого теста (секция закрыта, GATE-018)

**Для кого:** оператор вводного разговора и любой, кто читает этот атом
позже. Раздел был зарезервирован в `INPUT.md` ATOM-007 §3.4 как открытый —
живой сад-тест copilot-паттерна ещё шёл на момент формулирования. GATE-018
закрыл резерв: **двух доставленных находок достаточно**, ожидание (E1)
снято, дальнейшие находки сада входят отдельной поправкой, а не блокируют
это дерево.

## Находка №1 — as-is инвентаризация (источник: INFO-006, дословно)

> «...уже зафиксированная находка для intake: as-is инвентаризация
> обязательна для трансформационных задач (опись существующего + explicit
> preserve/change/remove до проектирования to-be).»

**Как повлияла на это руководство:** `INTAKE-GUIDE.md` §3 — целая ветка
разговора, обязательная до вывода целей (§4) при любом признаке
трансформационной задачи («у меня уже есть...»): опись существующего +
явный список «сохранить / изменить / убрать», вслух с фаундером, до того
как обсуждается новый продукт. Пропущена только для чистого листа
(greenfield).

## Находка №2 — невидимость команды (источник: INFO-007, дословно)

> «Полевой урок сад-теста №2 (в INPUT 007, секция «уроки полевого теста» +
> кандидат в touch-очередь): невидимость команды смазывает ценность — CEO
> прожил проект и не увидел, какие роли работали, в каких перспективах, где
> спорили. Расширить «витрину работы»: (а) строка себестоимости — как
> заказано; (б) TEAM-сводка при закрытии каждого родительского атома (рендер
> из status.yaml/вердиктов/subtree-учёта): роли и вклад одной строкой каждая,
> карта линз, синтез-конфликты с исходами («линза X хотела A, линза Y
> забраковала из-за B, синтез выбрал C»), возвраты verify; plain language,
> фаундер-читаемо; (в) правило-кандидат для touch: RESULT родителя обязан
> включать TEAM-сводку — инвариант, не фича интерфейса.»

**Как повлияла на это руководство:** `INTAKE-GUIDE.md` §6 — оператор
проговаривает фаундеру, явно и заранее, что тот увидит короткую сводку
понятным языком в конце каждого крупного этапа: кто работал, какие точки
зрения смотрели на задачу, где расходились и как решили, сколько было
доработок после проверки. Это ожидание задаётся во вводном разговоре, не
открывается фаундером случайно позже — сама формулировка (в) («TEAM-сводка
— инвариант, не фича») означает, что фаундер вправе ожидать её всегда, и
разговор обязан это заранее сказать.

**Что это значит для этого атома:** E1-ожидание, зарезервированное в

сада, если появятся, входят как отдельная, новая поправка — не переоткрывая
и не блокируя этот раздел задним числом.
